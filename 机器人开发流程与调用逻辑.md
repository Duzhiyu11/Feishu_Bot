## 系统概述

VIDO是一个基于飞书（Feishu）的企业级AI助手机器人，主要功能包括：
- **智能对话**：群聊和私聊场景下的自然语言交互
- **任务管理**：创建、查询、更新、删除任务
- **JIRA集成**：查询任务票、RC锁仓信息等
- **文档解析**：解析飞书文档为Markdown格式
- **画板解析**：解析飞书画板（思维导图）并支持节点查询
- **知识库检索**：基于RAG的知识库查询

**核心技术**：
- **MCP (Model Context Protocol)**：连接LLM和外部工具的桥梁，让LLM能够自动调用Python函数执行具体操作（如创建任务、查询JIRA等）
- **Function Calling**：LLM根据用户意图自动决定调用哪个工具，无需手动编程

def msg_recv(json_data):

    elif chat_type == "group":  # 群聊

### 2. executor.py - 核心执行器

**职责**：
- 消息预处理和意图识别
- 调用LLM生成回复
- 流式输出管理
- 卡片创建和更新
- 画板和文档解析

**关键方法**：

#### 2.1 `back_send_msg()` - 私聊消息处理

**流程**：
1. 处理@用户信息，替换占位符
2. 检测特殊命令（解析文档、解析画板、查询节点、推送文章）
3. 如果是一般对话，调用LLM生成回复
4. 创建流式卡片，实时更新回复内容

**特殊命令检测**：
- **解析文档**：检测"解析文档"关键词 + 飞书链接 → 调用`doc_manager.parse_and_save_doc()`
- **解析画板**：检测"解析画板"关键词 + 飞书链接 → 调用`board_manager.parse_board_from_link()`
- **查询节点**：检测"查询"关键词 + 画板查询格式 → 调用`board_manager.search_root_nodes()`
- **推送文章**：检测"推送"关键词 + 飞书链接 → 调用`JiraNotify.generate_article_card()`

#### 2.2 `back_reply_msg()` - 群聊消息处理

**流程**：
1. 提取消息内容和@信息
2. 维护群聊历史记录（`msg_record`）
3. 如果@了机器人：
   - 检测特殊命令（同私聊）
   - 调用`GroupReplyTask`生成回复
4. 如果没@机器人：
   - 调用`CheckNeedReplyTask`判断是否需要回复
   - 如果需要，调用`GroupReplyTask`生成回复

**群聊历史管理**：
```python
# 获取最近3天的群聊消息
self.msg_record[chat_id] = self.feishu_msg.get_group_msgs(chat_id, 60*60*24*3)

# 消息格式：时间|用户id|消息id|父消息id|消息内容
d_msg = f"{time_str}|{user_id}|{message_id}|{parent_id}|{msg}\n"
```

### 3. agent/exec_task.py - 任务执行器

**职责**：
- 解析任务名称，获取任务定义
- 格式化prompt（system prompt + user prompt）
- 调用LLM API
- 处理工具调用（Function Calling）

**关键流程**：

```python
def exec_task(task: str, **kwargs):
    # 1. 获取任务定义（如 GroupReplyTask）
    this_task = eval(task)
    
    # 2. 提取agent配置和prompt
    sys_prompt = this_task.agent.prompt.format(**kwargs)
    user_prompt = this_task.prompt.format(**kwargs)
    
    # 3. 获取工具定义（如果有）
    if hasattr(this_task, "tools"):
        function_definitions = jam_llm.get_function_definitions_sync(
            apply_tools=this_task.tools
        )
    
    # 4. 构建消息列表
    req_msg = [
        {"role": "system", "content": sys_prompt},
        {"role": "user", "content": user_prompt}
    ]
    
    # 5. 调用LLM
    llm_response = jam_llm.invoke(
        req_msg, 
        function_definitions=function_definitions,
        **llm_kwargs
    )
    
    return {
        "response": llm_response,
        "sys_prompt": sys_prompt,
        "user_prompt": user_prompt
    }
```

### 4. agent/task.py - 任务定义

**任务类型**：

#### 4.1 GroupReplyTask - 群聊回复任务

**Prompt结构**：
- 群聊上下文（`d_msg`）
- 用户信息映射（`user_info`）
- 知识库内容（`knowledge`）
- 发送者信息（`sender_info`）

**核心规则**：
- 任务创建：检测关键词 → 调用`create_task`工具
- 任务查询：检测查询意图 → 调用`get_task`工具
- JIRA查询：检测"查询"+"问题/开发/测试" → 调用`query_assignee_tasks`工具
- RC锁仓：检测锁仓相关关键词 → 调用相应工具

#### 4.2 UserReplyTask - 私聊回复任务

**特点**：
- 不包含群聊历史
- 任务默认负责人是当前用户
- 其他逻辑与群聊类似

#### 4.3 CheckNeedReplyTask - 判断是否需要回复

**规则**：
- 必须@了其他用户（不是机器人）
- 必须包含任务相关关键词
- 否则不回复

### 5. agent/mcp.py - MCP工具定义

#### 5.1 什么是MCP？

**MCP (Model Context Protocol)** 是一个标准协议，用于让LLM应用能够安全、标准化地调用外部工具和功能。在VIDO机器人中，MCP的作用是：

1. **工具定义**：通过`@mcp.mcp.tool`装饰器定义可被LLM调用的工具函数
2. **工具暴露**：将工具函数暴露给LLM，让LLM知道有哪些工具可用
3. **自动调用**：LLM根据用户意图自动决定调用哪个工具，无需手动编程
4. **标准化接口**：提供统一的工具调用接口，简化LLM与外部系统的集成

#### 5.2 MCP在VIDO中的工作流程

```
┌─────────────────────────────────────────────────────────┐
│  1. 初始化MCP服务器                                      │
│     mcp = JamMCP("My MCP Server")                       │
│     mcp.run()  # 启动MCP服务器                           │
└──────────────────┬──────────────────────────────────────┘
                   │
                   ▼
┌─────────────────────────────────────────────────────────┐
│  2. 定义工具函数（使用@mcp.mcp.tool装饰器）              │
│     @mcp.mcp.tool                                       │
│     def create_task(...):                               │
│         # 工具实现                                       │
└──────────────────┬──────────────────────────────────────┘
                   │
                   ▼
┌─────────────────────────────────────────────────────────┐
│  3. 将MCP URL传递给JamLLM                               │
│     jam_llm = JamLLM(mcp_url=mcp.mcp)                  │
│     # LLM现在知道有哪些工具可用                         │
└──────────────────┬──────────────────────────────────────┘
                   │
                   ▼
┌─────────────────────────────────────────────────────────┐
│  4. LLM调用时获取工具定义                                │
│     function_definitions = jam_llm.get_function_        │
│         definitions_sync(apply_tools=agent_tools)       │
│     # 返回工具的函数签名和描述                           │
└──────────────────┬──────────────────────────────────────┘
                   │
                   ▼
┌─────────────────────────────────────────────────────────┐
│  5. LLM根据用户意图自动调用工具                          │
│     用户："创建一个任务：明天完成报告"                    │
│     → LLM识别意图 → 调用create_task工具 → 返回结果      │
└─────────────────────────────────────────────────────────┘
```

#### 5.3 MCP的核心优势

1. **解耦设计**：工具定义与LLM调用分离，易于维护和扩展
2. **自动决策**：LLM自动决定何时调用哪个工具，无需硬编码规则
3. **类型安全**：通过函数签名和类型注解，确保参数正确传递
4. **标准化**：遵循MCP协议标准，兼容性更好

#### 5.5 工具调用流程详解

**完整流程**：

```python

用户："创建一个任务：明天完成报告"

# 2. executor.py 调用LLM
ret = executor.llm_call("UserReplyTask", body="创建一个任务：明天完成报告", ...)

# 3. exec_task.py 准备工具定义
function_definitions = jam_llm.get_function_definitions_sync(
    apply_tools=["create_task", "get_task", ...]
)
# 返回工具的函数签名、参数说明、返回值说明等

# 4. LLM接收prompt和工具定义

    {"role": "system", "content": sys_prompt},  # 包含工具使用规则
    {"role": "user", "content": user_prompt}    # 用户问题
]
llm_response = jam_llm.invoke(
    req_msg, 
    function_definitions=function_definitions  # 告诉LLM有哪些工具可用
)

# 5. LLM分析用户意图
# LLM识别到"创建任务"意图，决定调用create_task工具

# 6. LLM返回工具调用请求
# {
#   "tool_calls": [{
#     "name": "create_task",
#     "arguments": {
#       "content": "明天完成报告",
#       "end_time": "2025-01-30 18:00:00",
#       ...
#     }
#   }]
# }

# 7. JamLLM执行工具调用
# 调用mcp.mcp中定义的create_task函数

# 8. 工具执行并返回结果
# {
#   "success": True,
#   "task_id": 123,
#   "message": "任务创建成功"
# }

# 9. LLM根据工具结果生成最终回复
# "✅ 任务创建成功
#  **任务**：明天完成报告
#  **状态**：进行中
#  **截止时间**：2025-01-30 18:00:00"
```

#### 5.6 MCP工具定义示例

```python
from magic_jam import JamMCP

# 初始化MCP服务器
mcp = JamMCP("My MCP Server")
mcp.run()

# 定义工具函数
@mcp.mcp.tool
def create_task(
    content: str,           # 任务内容
    owner_id: str = None,   # 负责人ID
    end_time: str = None,   # 截止时间
    ...
) -> Dict[str, Any]:
    """
    创建任务
    
    Args:
        content: 任务内容描述（必填）
        owner_id: 任务负责人ID（可选）
        end_time: 任务截止时间，格式：YYYY-MM-DD HH:MM:SS（可选）
        ...
    
    Returns:
        dict: 包含success、task_id等字段的字典
    """
    # 工具实现逻辑
    db = next(JamMySQL().get_db())
    task = Task(content=content, owner_id=owner_id, ...)
    db.add(task)
    db.commit()
    
    return {
        "success": True,
        "task_id": task.id,
        "message": "任务创建成功"
    }
```

**关键点**：
- `@mcp.mcp.tool`装饰器将函数注册为MCP工具
- 函数的docstring会被LLM读取，用于理解工具用途和参数
- 函数的类型注解帮助LLM正确传递参数
- 返回值会被传递给LLM，用于生成最终回复

## 消息处理流程

### 私聊消息流程

```
用户发送消息
    │
    ▼
main.py: msg_recv() 接收消息
    │
    ▼
executor.back_send_msg()
    │
    ├─→ 检测特殊命令？
    │   ├─→ 解析文档 → doc_manager.parse_and_save_doc()
    │   ├─→ 解析画板 → board_manager.parse_board_from_link()
    │   ├─→ 查询节点 → board_manager.search_root_nodes()
    │   └─→ 推送文章 → JiraNotify.generate_article_card()
    │
    └─→ 一般对话
        │
        ├─→ 获取知识库内容 (get_knowledge)
        ├─→ 创建流式卡片 (create_flow_card)
        ├─→ 调用LLM (llm_call → UserReplyTask)
        └─→ 流式更新卡片 (update_flow_card)
```

### LLM调用流程

```
executor.llm_call(task, **kwargs)
    │
    ▼
agent.exec_task.exec_task(task, **kwargs)
    │
    ├─→ 解析任务定义 (eval(task))
    ├─→ 格式化prompt (system + user)
    ├─→ 获取工具定义 (function_definitions)
    ├─→ 调用LLM API (jam_llm.invoke)
    │   │
    │   ├─→ LLM返回文本 → 直接返回
    │   │
    │   └─→ LLM调用工具 → 执行工具 → 返回结果给LLM → LLM生成最终回复
    │
    └─→ 返回结果 {response, sys_prompt, user_prompt}
```

## LLM调用机制

### Prompt构建

**System Prompt**：
- 从`task.agent.prompt`获取
- 使用`format(**kwargs)`格式化
- 包含角色定义、规则说明、工具说明

**User Prompt**：
- 从`task.prompt`获取
- 包含上下文信息（群聊历史、用户信息、知识库内容）
- 包含当前用户问题

### 工具调用（Function Calling）与MCP

**MCP的核心作用**：MCP是连接LLM和外部工具系统的桥梁，让LLM能够自动调用Python函数来执行具体操作。

**完整流程**：

1. **MCP服务器初始化**：
   ```python
   # agent/mcp.py
   from magic_jam import JamMCP
   mcp = JamMCP("My MCP Server")
   mcp.run()  # 启动MCP服务器，提供工具调用接口
   ```

2. **工具定义**：在`agent/mcp.py`中使用`@mcp.mcp.tool`装饰器定义工具
   ```python
   @mcp.mcp.tool
   def create_task(content: str, owner_id: str, ...):
       """创建任务的工具函数"""
       # 执行创建任务逻辑
       db = next(JamMySQL().get_db())
       task = Task(content=content, owner_id=owner_id, ...)
       db.add(task)
       db.commit()
       return {"success": True, "task_id": task.id}
   ```

3. **MCP URL传递**：将MCP服务器URL传递给JamLLM
   ```python
   # agent/exec_task.py
   from .mcp import mcp
   jam_llm = JamLLM(mcp_url=mcp.mcp)  # 告诉LLM工具在哪里
   ```

4. **工具注册**：通过`jam_llm.get_function_definitions_sync()`获取工具定义
   ```python
   # 从task.py中获取工具列表
   agent_tools = ["create_task", "get_task", ...]
   
   # 获取工具的函数签名和描述
   function_definitions = jam_llm.get_function_definitions_sync(
       apply_tools=agent_tools
   )
   # 返回格式化的工具定义，包含：
   # - 工具名称
   # - 参数说明（从函数签名和docstring提取）
   # - 返回值说明
   ```

5. **LLM调用**：将工具定义传递给LLM
   ```python
   llm_response = jam_llm.invoke(
       req_msg,  # 包含system prompt和user prompt
       function_definitions=function_definitions  # LLM知道有哪些工具可用
   )
   ```

6. **LLM决策**：LLM根据用户意图和工具定义，自动决定是否调用工具
   - LLM分析用户消息："创建一个任务：明天完成报告"
   - LLM识别到"创建任务"意图
   - LLM查看可用的工具，发现`create_task`工具
   - LLM决定调用`create_task`工具

7. **工具执行**：JamLLM通过MCP协议调用工具函数
   ```python
   # JamLLM内部执行（简化示例）
   tool_result = mcp.call_tool("create_task", {
       "content": "明天完成报告",
       "end_time": "2025-01-30 18:00:00",
       ...
   })
   # 返回：{"success": True, "task_id": 123}
   ```

8. **结果处理**：LLM根据工具结果生成最终回复
   - LLM接收到工具执行结果
   - LLM根据结果生成自然语言回复
   - 回复："✅ 任务创建成功\n**任务**：明天完成报告\n..."

**关键优势**：
- **自动化**：LLM自动决定何时调用哪个工具，无需硬编码规则
- **类型安全**：通过函数签名和类型注解，确保参数正确传递
- **解耦设计**：工具定义与LLM调用逻辑分离，易于维护
- **标准化**：遵循MCP协议，兼容性更好

**示例**：
```python
# 1. 定义工具（agent/mcp.py）
@mcp.mcp.tool
def create_task(content: str, owner_id: str, ...):
    # 执行创建任务逻辑
    return {"success": True, "task_id": ...}

# 2. 在task.py中注册工具
class GroupReplyTask:
    agent = VIDOAgent
    tools = ["create_task", "get_task", ...]  # 工具列表

# 3. LLM自动调用
# 用户："创建一个任务：明天完成报告"
# → LLM识别意图
# → LLM查看tools列表，发现create_task工具
# → LLM通过MCP调用create_task工具
# → 工具执行，返回结果
# → LLM根据结果生成回复
```

### 流式输出

**实现**：
```python
# 1. 创建流式卡片
card_id = create_flow_card(element_id, save_id=save_id)

# 2. LLM返回流式响应
ret = llm_call(task, stream=True)
# ret["response"] 是一个生成器

# 3. 实时更新卡片
for chunk in ret["response"]:
    temp_send += chunk
    if time.time() - current_time > 2:  # 每2秒更新一次
        send_d += temp_send
        update_flow_card(card_id, element_id, send_d, sequence=i)
        i += 1
```

## 功能模块详解

### 1. 任务管理

**创建任务流程**：
1. 用户消息包含任务关键词（创建、设置、任务等）
2. LLM识别意图，调用`create_task`工具
3. 工具检查是否已存在类似任务
4. 如果不存在，创建新任务并保存到数据库
5. LLM生成格式化回复

**任务格式**：
```
**任务**：[任务内容]
**状态**：[进行中/已完成/已取消/已超时]
**负责人**：[负责人姓名]
**来源**：[群聊/私聊]
**截止时间**：[YYYY-MM-DD HH:MM:SS]
```

### 2. JIRA任务票查询

**查询流程**：
1. 用户消息："查询@张三的问题任务票清单"
2. LLM识别查询意图
3. 从`user_info`中提取被@用户的`open_id`和`jira_user_id`
4. 调用`query_assignee_tasks`工具
5. 工具查询JIRA API，返回任务票列表
6. LLM格式化结果并回复

**时间筛选支持**：
- "2025年10月后" → `created_after="2025-10-01"`

### 3. 画板解析与查询

**解析流程**：
1. 用户发送："解析画板 [飞书链接]"
2. 系统检测到"解析画板"关键词和链接
3. 调用`board_manager.parse_board_from_link()`
4. 使用飞书API获取画板节点数据
5. 构建节点结构（根节点、子节点关系）
6. 保存为JSON文件：`{文档标题}-{根节点名称}.json`

**查询流程**：
1. 用户发送："查询 [文件名-根节点]"
2. 系统检测到查询格式
3. 调用`board_manager.search_root_nodes()`搜索匹配的画板
4. 获取根节点的子节点
5. 创建交互式卡片展示节点树
6. 用户点击节点按钮，加载子节点

### 4. 文档解析

**解析流程**：
1. 用户发送："解析文档 [飞书链接]"
2. 系统检测到"解析文档"关键词和链接
3. 调用`doc_manager.parse_and_save_doc()`
4. 使用`FeishuDocParser`解析文档内容
5. 转换为Markdown格式
6. 保存到`parsed_docs/`目录

### 5. 知识库检索

**检索流程**：
```python
def get_knowledge(self, msg):
    fmea_rag = JamRAG()
    ret = fmea_rag.retriever2(collection="vido", query=msg)
    return ret
```

**用途**：
- 为LLM提供上下文知识
- 支持项目计划查询
- 支持技术问题解答
